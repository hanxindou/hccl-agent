# 差距分析

本文档分析赛题要求与当前项目实现之间的差距。结论是：当前项目具备早期原型骨架，但距离正式比赛交付仍处于起步阶段。短期应优先保证原型可运行、文档完整、模拟验证可追溯；随后再进入 C/C++ HCCL 插件和真实集合通信实现。

# 已具备能力

## 项目基础目录

赛题要求：需要提交可独立运行的 Agent 工程。

当前项目：已有 agent、skills、simulator、config、prompts、tests 等目录。

原因：目录命名与赛题中的 Agent、Skills、Prompt、模拟器方向基本一致，便于后续扩展。

差距：目录存在不等于工程完整，README、Prompt、tests 当前基本为空。

## Agent 编排雏形

赛题要求：Agent 需要具备硬件感知、算法生成、代码编译适配、迭代调优等能力。

当前项目：HCCLAgent 能初始化多个 Skill，并已串联集合通信原语校验、配置加载、拓扑推断、算法选择、优化评估、推荐解释和策略生成。

原因：已经有模块化编排结构，且具备一个可运行的最小 Agent 决策闭环，后续可以逐步把规则逻辑替换为可追溯的 Agent 流程。

差距：当前不是 LLM Agent，也没有代码生成、编译、测试、调优闭环。

## Skills 模块雏形

赛题要求：提交 Agent 能力清单 Skills。

当前项目：已有 AlgorithmSkill、TopologySkill、ConfigSkill、OptimizationSkill、StrategySkill、PerformanceModel。

原因：模块边界初步对应算法选择、拓扑判断、配置、优化、策略、评分。

差距：Skills 目前是简单规则函数，没有能力清单文档、Prompt 绑定和运行日志。

## 静态集群配置

赛题要求：模拟器或实机验证需要提供硬件参数、拓扑结构和配置说明。

当前项目：config/cluster.json 包含 cluster_name、nodes、topology、bandwidth_gbps、latency_ms、device_type、memory_gb、available_algorithms。

原因：具备最小配置输入，可作为模拟器参数来源。

差距：缺少链路矩阵、HCCS/RoCE/PCIe 类型、误码率、异构设备、NUMA、HBM、UB 等关键字段。

## 简单模拟器

赛题要求：允许自行通过模拟器完成成果验证，模拟器需能模拟通信特性、拓扑结构和硬件参数。

当前项目：Simulator 能根据 algorithm、topology、nodes、message_size_mb 估算 latency、bandwidth、score。

原因：已具备性能评估入口和评分输出。

差距：当前公式是占位逻辑，不够精准，也没有日志、验证流程和可复现配置。

## 算法排名

赛题要求：需要展示性能对比和算法优化结果。

当前项目：OptimizationSkill 能对候选算法按 score 排序，输出 best_algorithm、best_result、ranking。

原因：具备最小对比机制。

差距：score 未与真实性能指标或赛题评分标准校准。

# 部分具备能力

## 通信算法选择

赛题要求：针对小数据、中数据、大数据、稀疏场景选择不同算法，例如 Butterfly、PairWise、NHR、动态分块 Ring、Mesh、Fat-Tree 等。

当前项目：AlgorithmSkill 只根据 message_size_mb 分为小于 1MB、小于 512MB、大于等于 512MB 三档。

原因：已有按消息大小选择算法的规则，但粒度很粗。

差距：没有覆盖 <=64KB、>=1GB、稀疏场景、原语类型、拓扑、链路状态和设备异构。

## 集合通信原语识别

赛题要求：至少正确实现 3 种核心集合通信原语。

当前项目：HCCLAgent.run() 已增加 primitive 参数，并支持 AllReduce、AllGather、ReduceScatter 三种原语名称校验。

原因：Agent 已经能明确当前优化目标属于哪一种集合通信原语，为后续分别实现原语逻辑、测试和报告打基础。

差距：当前只是原语识别和输入校验，还没有真实 AllReduce、AllGather、ReduceScatter 数据通信实现，也没有让 primitive 影响算法选择、模拟器公式和通信策略。

## 拓扑感知

赛题要求：调用 hcclGetTopology 获取 NPU 物理连接信息，构建加权有向图，建模最优通信路径和链路拥塞。

当前项目：TopologySkill 可根据 nodes 返回 Full Mesh、Ring、Fat Tree，且已接入 HCCLAgent.run() 主流程。

原因：具备拓扑分类雏形，并已能让拓扑影响模拟器评估结果。

差距：没有真实探测、没有图结构、没有链路权重、没有拥塞预测。

## 性能建模

赛题要求：采集和分析带宽、延迟、缓存命中率、链路利用率、NPU 利用率、训练耗时等指标。

当前项目：PerformanceModel 用 bandwidth * 0.7 - latency * 0.3 计算 score。

原因：已具备 latency、bandwidth、score 三个基础指标。

差距：没有缓存、链路利用率、端到端训练指标；单位不统一；公式缺少实验依据。

## 通信策略输出

赛题要求：Agent 生成具体算法代码和通信调度逻辑。

当前项目：StrategySkill 能为 Ring 输出环形路径，为 Butterfly 输出固定配对步骤。

原因：已有通信步骤文本化表达。

差距：不是可执行通信代码；Butterfly 固定为 8 节点；没有原语语义和数据分块。

## Agent 专项工程

赛题要求：提交完整 Agent 工程、能力清单、Prompt 文件、依赖库清单、执行脚本和可复现日志。

当前项目：有 agent/skills/prompts 目录，Agent 主流程已能输出推荐算法、推荐原因、模拟结果和通信策略。

原因：工程命名和模块方向与赛题一致。

差距：prompts/algorithm_prompt.txt 为空，没有 LLM 调用、没有日志、没有依赖清单、没有执行脚本。

## 最小自动化测试

赛题要求：提交测试用例，并证明功能正确性可复现。

当前项目：tests/test_agent.py 已覆盖 Agent 对 AllReduce、AllGather、ReduceScatter 三个原语的识别、非法原语拒绝逻辑，以及 main.py CLI 参数解析。

原因：这是从空测试目录到可复现验证的第一步，能防止后续修改破坏 Agent 的基础入口。

差距：当前测试还没有验证真实集合通信输出是否与 HCCL 基准一致，也没有覆盖混合精度、性能、压测和故障注入。

## 非交互式运行入口

赛题要求：提交可独立运行的 Agent 工程，并保证验证流程可追溯、可复现。

当前项目：main.py 已支持 --nodes、--message-size、--primitive 参数，可以不用手动输入就复现实验。

原因：非交互式入口便于后续编写测试脚本、性能验证脚本和 Agent 运行日志采集。

差距：当前还没有将每次运行结果写入结构化日志文件，也没有形成完整验证报告。

## 模拟器验证

赛题要求：模拟器需精准模拟昇腾 NPU 集群通信特性，提交配置说明、验证流程、性能参数和完整日志。

当前项目：有 simulator/simulator.py 和 config/cluster.json。

原因：已有模拟器入口和静态配置。

差距：缺少准确模型、验证用例、日志输出、配置说明和与实机标准的一致性说明。

# 完全缺失能力

## C/C++ HCCL 插件

赛题要求：代码采用 C/C++ 开发，遵循 HCCL 插件接口，生成 .so 动态库。

当前项目：没有 C/C++ 源码、头文件、CMake、动态库。

原因：当前是 Python 概念原型。

影响：这是 P0 级差距，不补齐就无法满足正式交付。

## CANN/HCOMM 接口接入

赛题要求：基于 CANN 8.0 或之后版本，只能基于 HCOMM 开源仓接口开发。

当前项目：没有 CANN、HCOMM、hcclCommInit、hcclAllReduce、hcclGetTopology 等接口调用。

原因：当前没有引入昇腾 SDK 或接口适配层。

影响：无法证明昇腾环境兼容性。

## 至少 3 种真实集合通信原语

赛题要求：至少正确实现 3 种核心集合通信原语。

当前项目：Agent 已能识别 AllReduce、AllGather、ReduceScatter 三个原语名称，但只有算法名称和模拟评分，没有这些原语的真实数据逻辑。

原因：刚完成原语参数入口，尚未进入通信原语实现阶段。

影响：无法通过正确性评判。

## 正确性验证

赛题要求：输出结果与原生 HCCL 基准一致，支持混合精度，无精度漂移。

当前项目：已有最小 Agent 入口测试，但没有通信结果断言、基准对比或精度验证。

原因：测试体系刚开始建立，当前只覆盖 Agent 输入输出结构。

影响：无法证明功能正确。

## 可靠性机制

赛题要求：链路故障自愈、数据校验、超时重传、流量控制、健康检测、动态路由切换、重传率统计。

当前项目：没有故障模型和可靠性逻辑。

原因：模拟器和策略层尚未建模链路状态。

影响：无法完成可靠性报告和故障注入演示。

## 压测与 Profiling

赛题要求：压测脚本、72 小时高并发压测、msprof 或模拟器采集性能指标、可视化分析。

当前项目：没有压测脚本、Profiling 数据采集、报告生成。

原因：项目仍停留在单次公式评估。

影响：无法支撑性能报告。

## 硬件深度适配

赛题要求：适配 HCCS、RoCE、PCIe、NUMA、HBM、UB、随路归约、零 DDR 拷贝、计算通信重叠。

当前项目：没有这些硬件概念的模型或代码。

原因：cluster.json 只包含基础字段，Simulator 也未使用硬件细节。

影响：算法缺少昇腾特色，难以获得高分。

## 异构集群支持

赛题要求：支持混合 910A2/910A3、不同代际 NPU、非对称带宽链路。

当前项目：device_type 是单个字符串 Ascend910A。

原因：配置结构不支持多设备和链路矩阵。

影响：无法覆盖异构场景。

## 动态拓扑自适应

赛题要求：支持节点增减、动态拓扑变化和算法自动重构。

当前项目：拓扑是静态字符串，节点数来自用户输入和配置文件。

原因：没有运行时拓扑事件和重规划机制。

影响：无法覆盖极限场景。

## Agent 代码生成可复现

赛题要求：核心算法及代码通过 Agent 完成，并提交 Prompt 调用记录和 Agent 运行日志。

当前项目：没有 LLM 调用、没有代码生成记录、没有日志。

原因：当前 Agent 只是 Python 类编排。

影响：Agent 专项评分风险很高。

## 交付材料

赛题要求：算法设计说明书、性能测试报告、可靠性报告、模拟器说明、验证日志、5 分钟演示视频。

当前项目：本次新增了初始赛题分析、项目文档和差距分析，但正式交付材料尚未形成。

原因：项目处于基线梳理阶段。

影响：后续每次开发必须同步沉淀实验和文档。
