# 赛题目标

赛题名称为“基于 CANN 的分布式通信算法设计与优化”。核心目标是面向昇腾 NPU 集群，包括单机多卡和多机多卡场景，依托智能 Agent 技术完成 HCCL 集合通信算法的设计、代码开发、验证和调优。

赛题想解决的问题是大模型分布式训练中的通信瓶颈。随着模型规模从百亿、千亿迈向万亿级，AllReduce、AllGather、ReduceScatter、Broadcast、AlltoAll 等集合通信在训练迭代中的占比显著升高，通信开销会限制集群扩展性、算力利用率和训练收敛速度。因此作品需要围绕“算法-拓扑-硬件”协同优化，提升带宽、降低延迟，并保证正确性、可靠性和可复现性。

赛题特别强调 Agent 的作用：核心算法设计和代码开发应通过 Agent 完成，提交物中必须能展示 Agent 的 Skills、Prompt 工程、代码生成过程、验证流程和运行日志。

# 评分点分析

赛题文档未给出明确百分比分值，但评判标准和作品要求可以整理为以下评分方向。

## 通信原语正确性

重要程度：最高。

要求至少正确实现 3 种核心集合通信原语，例如 AllReduce、AllGather、ReduceScatter、Broadcast。输出结果需要与昇腾原生 HCCL 基准结果数值一致，支持 FP32、BF16、FP16 混合精度，无 NaN、溢出、下溢或精度漂移。

这是进入有效评测的基础。如果原语不正确，性能优化没有意义。

## 昇腾环境兼容性

重要程度：最高。

要求基于 CANN 8.0 或之后版本，只能基于 HCOMM 开源仓提供的接口开发，遵循 HCCL 插件接口，例如 hcclCommInit、hcclAllReduce 等。代码需要采用 C/C++，可编译、可运行、可链接，不依赖私有 API，不修改底层驱动或链路固件。

当前项目是 Python 原型，与该要求差距很大。后续 P0 阶段必须建立 C/C++ 插件骨架与可编译链路。

## 通信性能

重要程度：最高。

评判围绕带宽、延迟、加速比、链路利用率、NPU 利用率、端到端训练耗时等指标。赛题期望在单机 8 卡、多机多卡、8 到 1024 卡扩展场景中取得高吞吐、低延迟和较高线性加速比。

模拟器验证结果可作为性能评判的重要参考，与实机验证同等对待，但需要配置说明、验证流程、性能参数和完整日志。

## 算法创新性

重要程度：高。

赛题不希望只停留在标准 Ring、Mesh、HD 算法，而是鼓励自适应拓扑算法、NHR 非均匀环、分块 Mesh、动态 Butterfly、稀疏通信、量化压缩、通算融合等方向。

创新需要建立在正确性和可验证性之上。对于本项目，应先实现基线算法，再逐步加入拓扑感知和分块策略。

## 硬件感知与拓扑适配

重要程度：高。

要求 Agent 或生成代码能够感知 HCCS、RoCE、PCIe 的带宽、延迟、误码率等参数，感知 NPU NUMA 拓扑、HBM 分区、片上缓存 UB 容量等硬件信息，并基于拓扑构建加权图、最优路径、链路拥塞预测等模型。

这是与普通通信算法项目拉开差距的关键评分点。当前项目只有静态 JSON 配置和简单拓扑字符串，属于早期占位。

## 可靠性

重要程度：高。

要求支持链路故障自愈、数据校验、超时重传、流量控制、健康检测、动态路由切换等机制。文档中提到主链路故障时 100ms 内切换备用路径、CRC32 加奇偶校验、重传率不超过 0.1% 等方向。

可靠性会影响千卡场景稳定性，也是最终报告和演示的重要内容。

## Agent 专项输出物

重要程度：高。

必须提交完整 Agent 工程、运行环境配置、依赖清单、执行脚本、Agent 能力清单、Prompt 文件、Prompt 调用记录、Agent 运行日志，确保评委可以复现 Agent 生成算法和代码的全过程。

本项目当前目录中已有 agent、skills、prompts 的雏形，但尚未形成真实 LLM/Agent 生成闭环。

## 工程化与文档

重要程度：中高。

要求提交代码包、头文件、编译脚本 CMake、测试用例、压测脚本、故障注入工具、算法设计说明、性能测试报告、可靠性报告、模拟器说明和演示视频。

工程化不足会影响复现和可信度。后续每次开发都必须同步更新文档，保持代码、实验和说明一致。

# 关键技术点

## HCCL

Huawei Collective Communication Library，是赛题最终交付的核心接口目标。作品需要围绕 HCCL 集合通信原语实现插件或兼容代码。

## HCOMM / CANN

赛题要求基于 CANN 8.0 或之后版本，并基于 HCOMM 开源仓提供的接口开发。后续代码应逐步从 Python 原型过渡到可编译的 C/C++ HCCL 插件工程。

## 集合通信原语

至少覆盖 3 种核心原语。优先级建议为：

1. AllReduce：分布式训练梯度同步，最核心。
2. AllGather：模型并行参数聚合。
3. ReduceScatter：数据并行分片归约。
4. Broadcast：参数初始化广播，可作为补充。
5. AlltoAll：专家并行和序列并行，复杂度更高，可作为后续优化。

## 拓扑感知

需要从静态拓扑字符串发展为可建模的拓扑图，包含节点、链路类型、带宽、延迟、拥塞、故障状态等信息。算法选择和通信路径应基于拓扑，而不是只基于节点数或消息大小。

## 通信优化

包括 Ring、Butterfly、Mesh、NHR、分层算法、分块调度、稀疏通信、量化压缩、通信计算重叠等。短期目标应先有可验证基线，再进行场景化优化。

## 性能建模

模拟器需要建模延迟、带宽、消息大小、节点规模、拓扑、链路类型、算法复杂度等因素。当前项目已有一个简单 score 函数，但与真实通信性能模型差距较大。

## Agent

Agent 不应只做静态规则选择。赛题要求 Agent 能完成硬件感知、算法生成、代码生成、编译适配、测试生成、性能分析和迭代调优，并保留可复现日志。

## LLM 与 Prompt 工程

Prompt 需要覆盖算法设计、代码生成、测试用例生成、性能分析、可靠性机制生成等任务。当前 prompts/algorithm_prompt.txt 为空，需要补齐。

## 模拟器

赛题允许用模拟器验证成果，但要求精准模拟昇腾 NPU 集群通信特性、拓扑结构和硬件参数，并提交配置说明、验证流程和完整日志。
